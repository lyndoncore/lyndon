Selector Matching Performance

Although writing efficient CSS selectors is not as critial to the web performance landscape as it once was, (see Nicole Sullivans article on CSS selector performance), selector matching still plays a considerable role in large stylesheets. Calculating the style of a page entails the browser matching CSS class/ID/tag/universal rules with elements specified within our HTML. The more complex these CSS selectors are, the more expensive the operation of selector matching becomes for the browser. Intuitively, this means we should probably stick to simpler class names when writing css, avoiding unnessecary work for the browser. This can be a worthwhile performance consideration, especially when there are several great CSS naming conventions like SMACSS(link) and BEM(link).

I've weighed the cost of relative values

Working with relative values can make responsive design very easy, but at the same time, defers work from the developer to the browser. In order for the browser to enevitably paint pixels to the screen, it must first make sense of a relative value (like "3rem" or "50%") by translating it to a computed value - something the browser can work with. On the other hand, absolute values (like "14px" or "blue") do not require the broswer to do any interpretation. In a sense, absolute values are delivered ready-to-use for the browser. The decision between using relative and absolute values, is a design decision that should be measured and considered when optimizing for frontend performance. https://www.w3.org/TR/1998/REC-CSS2-19980512/cascade.html
